MADCSS philosophy

MADCSS stands for Modular Attribute-Driven CSS. Yeah, sooo clever. Basic architecture is based on inspired by: AMCSS, Gemma, Organic CSS, Atomic Design and Tachyons.

The base global prefix for all attribute modules is i-.

Naming things

Attribute names are defined in a simple fashion: attribute always starts with the global prefix, followed by module name. If it's a child, then it appends a single hyphen, followed by the child name. Note: you shouldn't go deeper than 2 levels: i.e. grandchildren and further are forbidden.

{{ "-[-]" | escape }}

Modifiers are a space-separated strings, so basically you can use any character you want:

{{ '
…
' | escape }}

And the you would write this in your .css file:

/* nietzsche module */
[i-nietzsche] { … }
[i-nietzsche~="2/3"] { … }
[i-nietzsche~="isDead"] { … }
[i-nietzsche~="+_-"] { … }
[i-nietzsche~="żółć"] { … }
[i-nietzsche~="鸡汤"] { … }

Architecture

There are not a lot of files. You will basically create new things in modules/ directory and some global variables in -- wait for it -- variables.css!

modules/
    nietzsche.css
setup/
    root.css
    states.css
    typography.css
main.css
variables.css

As for variables, make sure that you define module own vars in the :root {} block on top of the file. If multiple modules use a variable, you should put in in global file. And the thing I am passionately crazy about: define new colors only inside variables.css -- this assures your design is not "50 shades of gray".

The setup/ dir contains some resets for the whole project and simple typography normalisations.

Modules can be either structure- and layout-related selectors that do one, generic thing extremely well -- or a context-specific selectors for distinct components of an interface. Can be naturally mixed with each other -- as long as it makes sense of course. Try to avoid "Swiss Army knife" modules, as it will be hell to maintain them.
